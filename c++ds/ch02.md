# 2. Foundations for Building Data Structures in C++

\index{foundations}

Before diving into specific data structures, you need a solid grasp of the C++ tools that make them possible.
This chapter reviews the mechanics of classes and templates, dynamic memory, RAII, constructors and destructors, ownership semantics, error handling, and algorithmic complexity.
Some of this material was covered in *Gorgo Continuing C++* --- this chapter sharpens your understanding with a data-structures focus so you are ready to implement containers that are correct, efficient, and resource-safe.

## Classes and Templates

\index{class!review}
\index{template!review}

A class groups related data and operations.
A **template** parameterizes a class (or function) on a type, letting you write the logic once and reuse it for any type.

```cpp
#include <iostream>
#include <string>

template<typename T>
class Box {
public:
    explicit Box(const T& value) : value_{value} {}

    const T& get() const { return value_; }
    void set(const T& value) { value_ = value; }

private:
    T value_;
};

int main()
{
    Box<int> n{42};
    Box<std::string> s{"1984"};

    std::cout << n.get() << "\n";
    std::cout << s.get() << "\n";

    return 0;
}
```

```
42
1984
```

Templates are resolved at compile time --- the compiler generates a separate `Box<int>` and `Box<std::string>` class from the single template definition.
There is no runtime cost for the abstraction.

### Member Functions Outside the Class

When a template class has member functions defined outside the class body, the template parameter must be repeated:

```cpp
template<typename T>
class Box {
public:
    explicit Box(const T& value);
    const T& get() const;
private:
    T value_;
};

template<typename T>
Box<T>::Box(const T& value) : value_{value} {}

template<typename T>
const T& Box<T>::get() const { return value_; }
```

::: {.tip}
**Tip:** For templates, keep definitions in header files (`.h` or `.hpp`).
The compiler needs to see the full template definition when it instantiates it, so putting it in a `.cpp` file will cause linker errors in other translation units.
:::

## Dynamic Memory and Resource Management

\index{dynamic memory}
\index{RAII}

Data structures often need to allocate memory at runtime --- you do not always know at compile time how many elements will be stored.
Dynamic memory (`new`/`delete`) lets you allocate on the heap, but it introduces the problem of ownership: who is responsible for freeing the memory?

The C++ answer is **RAII** (Resource Acquisition Is Initialization): tie the lifetime of a resource to the lifetime of an object.
When the object is constructed, it acquires the resource; when it is destroyed, it releases it.

```cpp
#include <iostream>
#include <stdexcept>
#include <string>

// A simple RAII wrapper for a heap-allocated array
template<typename T>
class HeapArray {
public:
    explicit HeapArray(int capacity)
        : data_{new T[capacity]}, capacity_{capacity}, size_{0}
    {}

    ~HeapArray()
    {
        delete[] data_;
    }

    void append(const T& item)
    {
        if (size_ >= capacity_) {
            throw std::overflow_error("array is full");
        }
        data_[size_++] = item;
    }

    T& operator[](int index) { return data_[index]; }
    const T& operator[](int index) const { return data_[index]; }

    int size() const { return size_; }

private:
    T* data_;
    int capacity_;
    int size_;
};

int main()
{
    HeapArray<std::string> books{3};
    books.append("1984");
    books.append("Brave New World");

    for (int i = 0; i < books.size(); ++i) {
        std::cout << books[i] << "\n";
    }

    return 0;
}
```

```
1984
Brave New World
```

When `books` goes out of scope, the destructor automatically calls `delete[]`.
There is no memory leak even if an exception is thrown --- RAII handles it.

::: {.tip}
**Trap:** Mixing `new` and `delete[]` (or `new[]` and `delete`) is undefined behavior.
Always pair `new` with `delete` and `new[]` with `delete[]`.
:::

### `std::unique_ptr` for Single Ownership

\index{unique\_ptr}

For single-ownership resources, prefer `std::unique_ptr` over raw `new`/`delete`:

```cpp
#include <memory>
#include <iostream>

int main()
{
    auto arr = std::make_unique<int[]>(5);
    for (int i = 0; i < 5; ++i) {
        arr[i] = i * i;
    }
    for (int i = 0; i < 5; ++i) {
        std::cout << arr[i] << " ";
    }
    std::cout << "\n";
    return 0;
}
```

```
0 1 4 9 16
```

`unique_ptr` automatically calls `delete[]` when it goes out of scope --- no manual cleanup needed.

## Constructors, Destructors, and Copy Control

\index{constructor}
\index{destructor}
\index{copy control}

When a class manages a resource (like heap memory), you must define --- or explicitly delete --- the **Big Five**: copy constructor, copy assignment, move constructor, move assignment, and destructor.
If you leave any of them as defaults when the class owns a resource, you will get shallow copies and double-free bugs.

```cpp
#include <algorithm>
#include <iostream>
#include <string>

template<typename T>
class Buffer {
public:
    explicit Buffer(int cap)
        : data_{new T[cap]}, size_{0}, capacity_{cap}
    {}

    // Destructor
    ~Buffer() { delete[] data_; }

    // Copy constructor
    Buffer(const Buffer& other)
        : data_{new T[other.capacity_]},
          size_{other.size_},
          capacity_{other.capacity_}
    {
        std::copy(other.data_, other.data_ + other.size_, data_);
    }

    // Copy assignment
    Buffer& operator=(const Buffer& other)
    {
        if (this == &other) return *this;
        T* new_data = new T[other.capacity_];
        std::copy(other.data_, other.data_ + other.size_, new_data);
        delete[] data_;
        data_ = new_data;
        size_ = other.size_;
        capacity_ = other.capacity_;
        return *this;
    }

    // Move constructor
    Buffer(Buffer&& other) noexcept
        : data_{other.data_}, size_{other.size_}, capacity_{other.capacity_}
    {
        other.data_ = nullptr;
        other.size_ = 0;
        other.capacity_ = 0;
    }

    // Move assignment
    Buffer& operator=(Buffer&& other) noexcept
    {
        if (this == &other) return *this;
        delete[] data_;
        data_ = other.data_;
        size_ = other.size_;
        capacity_ = other.capacity_;
        other.data_ = nullptr;
        other.size_ = 0;
        other.capacity_ = 0;
        return *this;
    }

    void push(const T& item)
    {
        if (size_ < capacity_) {
            data_[size_++] = item;
        }
    }

    int size() const { return size_; }
    const T& operator[](int i) const { return data_[i]; }

private:
    T* data_;
    int size_;
    int capacity_;
};

int main()
{
    Buffer<std::string> a{3};
    a.push("Fahrenheit 451");
    a.push("Brave New World");

    Buffer<std::string> b = a;  // copy
    b.push("1984");

    std::cout << "a size: " << a.size() << "\n";
    std::cout << "b size: " << b.size() << "\n";
    std::cout << "b[2]: " << b[2] << "\n";

    return 0;
}
```

```
a size: 2
b size: 3
b[2]: 1984
```

Without the copy constructor, `b = a` would copy the *pointer*, not the data.
Both `a` and `b` would point to the same array, and when either destructor ran, the other would have a dangling pointer.

::: {.tip}
**Tip:** The **Rule of Five**: if you define any of destructor, copy constructor, copy assignment, move constructor, or move assignment, you should define or explicitly delete all five.
If you define a destructor, the compiler-generated copy/move operations are likely wrong.
:::

## Pointers, References, and Ownership

\index{pointer!ownership}
\index{reference}

Data structure nodes often reference other nodes.
Getting ownership right is critical.

- A **raw pointer** (`T*`) that *owns* its target is dangerous --- you must manually `delete` it.
  Reserve owning raw pointers for internal implementation details where you have complete control.
- A `std::unique_ptr<T>` expresses exclusive ownership.
  When the `unique_ptr` is destroyed, the target is too.
- A `std::shared_ptr<T>` expresses shared ownership.
  The target lives until the last `shared_ptr` pointing to it is destroyed.
- A **raw pointer** or **reference** used as a *non-owning observer* is fine --- it does not manage the lifetime of what it points to.

In linked data structures you will typically use `unique_ptr` for ownership (each node owns its successor) and raw pointers for non-owning traversal:

```cpp
#include <iostream>
#include <memory>
#include <string>

struct Node {
    std::string value;
    std::unique_ptr<Node> next;

    explicit Node(const std::string& v) : value{v}, next{nullptr} {}
};

int main()
{
    auto head = std::make_unique<Node>("1984");
    head->next = std::make_unique<Node>("Brave New World");
    head->next->next = std::make_unique<Node>("Fahrenheit 451");

    // Non-owning traversal
    for (const Node* p = head.get(); p != nullptr; p = p->next.get()) {
        std::cout << p->value << "\n";
    }

    return 0;
}
```

```
1984
Brave New World
Fahrenheit 451
```

When `head` goes out of scope, the entire chain is deleted automatically because each node owns the next via `unique_ptr`.

## Error Handling and Defensive Programming

\index{error handling}
\index{precondition!checking}

Data structures have preconditions.
When a caller violates them (e.g., calling `pop()` on an empty stack), you have two main options:

1. **Throw an exception** to report the error and let the caller handle it.
2. **Assert and abort** to catch programming errors early in development.

```cpp
#include <cassert>
#include <stdexcept>

void pop_example(int* data, int& top)
{
    // Option 1: exception (good for expected error conditions)
    if (top < 0) {
        throw std::underflow_error("pop on empty stack");
    }

    // Option 2: assert (good for catching programmer mistakes in debug builds)
    assert(top >= 0 && "pop on empty stack");

    --top;
}
```

For data structures that are used as internal implementation details, `assert` is often appropriate --- it documents the precondition and is compiled out in release builds with `-DNDEBUG`.
For data structures with public APIs, `throw` is better because the caller can catch and handle the error.

::: {.tip}
**Tip:** Use `assert` to document and check preconditions in internal code.
Use `throw` to report recoverable errors to callers.
Do not use `assert` in release code for conditions that depend on user input or external data.
:::

## Measuring Efficiency: Big-O Review

\index{Big-O notation}
\index{time complexity}

Every operation in a data structure has a **time complexity** --- how the number of steps grows as the input size grows.
**Big-O notation** describes the dominant term:

| Notation | Name | Example |
|----------|------|---------|
| O(1) | Constant | Array index access |
| O(log n) | Logarithmic | Binary search |
| O(n) | Linear | Scanning an array |
| O(n log n) | Linearithmic | Merge sort |
| O(n²) | Quadratic | Bubble sort |
| O(2ⁿ) | Exponential | Naïve recursive Fibonacci |

Big-O drops constants and lower-order terms.
An operation that takes `3n + 7` steps is O(n).
An operation that takes `2n² + 100n` is O(n²).

### Space Complexity

\index{space complexity}

Space complexity measures how much extra memory an algorithm uses.
An in-place sort that uses a fixed amount of extra memory is O(1) space.
An algorithm that creates a copy of the input is O(n) space.

### Amortized Complexity

\index{amortized complexity}

Some operations are occasionally expensive but cheap on average.
A growable array (`std::vector`) doubles its capacity when full.
Most appends are O(1), but the occasional resize copies all elements in O(n).
If you spread the cost of the resize over all the appends that led to it, each append costs O(1) on average --- this is **amortized O(1)**.

::: {.tip}
**Wut:** O(1) amortized does not mean "always O(1)."
It means the *average* cost per operation is O(1) over a long sequence.
A single operation can still be O(n) if it triggers a resize.
:::

### Which Operations Matter?

When you choose a data structure, identify which operations your workload calls most.
If you mostly insert at the front and rarely search, a linked list beats an array.
If you mostly search by index and rarely insert, an array beats a linked list.
Big-O analysis is your tool for making that decision quantitatively.

## Try It: Building a Growable Buffer

This starter program implements a growable buffer with the RAII and Big Five patterns.
Type it in, compile, and experiment with it:

```cpp
#include <algorithm>
#include <iostream>
#include <stdexcept>
#include <string>

template<typename T>
class GrowableBuffer {
public:
    explicit GrowableBuffer(int initial_cap = 4)
        : data_{new T[initial_cap]}, size_{0}, capacity_{initial_cap}
    {}

    ~GrowableBuffer() { delete[] data_; }

    GrowableBuffer(const GrowableBuffer& other)
        : data_{new T[other.capacity_]},
          size_{other.size_},
          capacity_{other.capacity_}
    {
        std::copy(other.data_, other.data_ + other.size_, data_);
    }

    GrowableBuffer& operator=(GrowableBuffer other)
    {
        std::swap(data_, other.data_);
        std::swap(size_, other.size_);
        std::swap(capacity_, other.capacity_);
        return *this;
    }

    GrowableBuffer(GrowableBuffer&& other) noexcept
        : data_{other.data_}, size_{other.size_}, capacity_{other.capacity_}
    {
        other.data_ = nullptr;
        other.size_ = 0;
        other.capacity_ = 0;
    }

    void push_back(const T& item)
    {
        if (size_ == capacity_) {
            grow();
        }
        data_[size_++] = item;
    }

    int size() const { return size_; }
    int capacity() const { return capacity_; }

    const T& operator[](int i) const
    {
        if (i < 0 || i >= size_) throw std::out_of_range("index out of range");
        return data_[i];
    }

private:
    void grow()
    {
        int new_cap = capacity_ * 2;
        T* new_data = new T[new_cap];
        std::copy(data_, data_ + size_, new_data);
        delete[] data_;
        data_ = new_data;
        capacity_ = new_cap;
    }

    T* data_;
    int size_;
    int capacity_;
};

int main()
{
    GrowableBuffer<std::string> buf{2};

    for (const std::string& title : {"1984", "Brave New World", "Fahrenheit 451"}) {
        buf.push_back(title);
        std::cout << "size=" << buf.size() << " cap=" << buf.capacity() << "\n";
    }

    for (int i = 0; i < buf.size(); ++i) {
        std::cout << buf[i] << "\n";
    }

    return 0;
}
```

```
size=1 cap=2
size=2 cap=2
size=3 cap=4
1984
Brave New World
Fahrenheit 451
```

Try triggering another resize by adding a fourth and fifth book.
Try copying the buffer and modifying the copy to verify they are independent.

## Key Points

- **Templates** generate type-specific code at compile time with zero runtime overhead.
  Keep template definitions in header files.
- **RAII** ties resource lifetime to object lifetime: acquire in the constructor, release in the destructor.
- Define the **Big Five** (destructor, copy constructor, copy assignment, move constructor, move assignment) whenever your class owns a resource.
- Use `unique_ptr` for exclusive ownership and raw (non-owning) pointers for traversal.
- Use `assert` to document internal preconditions; use `throw` to report recoverable errors to callers.
- **Big-O notation** drops constants and lower-order terms and describes how work scales with input size.
- **Amortized O(1)** means the average cost per operation is O(1) over a sequence, even if individual operations are occasionally expensive.
- Choose a data structure by matching its Big-O characteristics to your workload's dominant operations.
