# 9. Comparing Data Structures

\index{data structure!comparison}

You have now studied arrays, linked lists, stacks, queues, heaps, and hash tables.
Each solves a different problem well and performs poorly at others.
This chapter pulls them together into a single reference: a comparison of operations, trade-offs, and selection criteria.
By the end you should be able to look at a problem and immediately narrow down which data structures are candidates.

## Summary Table

Where two complexities appear, the first is average case and the second is worst case.

|-----------|-------------|--------|-------------|------------|--------------|-------------|------------|--------------|-------|

| Growable array | O(1) | O(n) | O(n) | O(1)* | O(n) | O(n) | O(1) | O(n) | O(n) |
| Singly linked list | O(n) | O(n) | O(1) | O(1)** | O(n) | O(1) | O(n) | O(n) | O(n) |
| Doubly linked list | O(n) | O(n) | O(1) | O(1) | O(1)*** | O(1) | O(1) | O(1)*** | O(n) |

## Choosing the Right Structure

\index{data structure!selection criteria}

Here are the decision criteria that matter most in practice:

### Need fast access by key?

Use a **hash table** (`std::unordered_map`).
If you also need keys in sorted order, use a **BST** (`std::map`).

### Need fast access by index?

Use a **growable array** (`std::vector`).
Nothing else comes close for random access.

### Need fast insert/delete at the front?

Use a **linked list** or **deque** (`std::deque`).
Prepending to a `std::vector` is O(n) because every element must shift.

### Need the maximum or minimum efficiently?

Use a **heap** or `std::priority_queue`.
Finding the max in an unsorted array is O(n); a heap always gives it in O(1).

### Need LIFO ordering?

Use a **stack** (`std::stack`).

### Need FIFO ordering?

Use a **queue** (`std::queue`).

### Need to insert and delete at both ends?

Use a **deque** (`std::deque`) or a doubly linked list.

## Cache Behavior and Real-World Performance

\index{cache!performance}

Big-O analysis ignores constant factors, but cache behavior can dominate in practice.

Modern CPUs load memory in 64-byte cache lines.
If you access an array sequentially, each cache line brings in 16 `int` values at once --- the next 15 accesses are essentially free.
A linked list, by contrast, scatters nodes across the heap.
Each node access is likely a cache miss --- roughly 200× slower than a cache hit on modern hardware.

This is why `std::vector` consistently outperforms `std::list` for sequential workloads, even though they have the same asymptotic complexity for traversal.
It is also why the circular-array queue outperforms the linked-list queue for most practical workloads.

::: {.tip}
**Tip:** When you profile a program and find a data structure bottleneck, try switching to a more cache-friendly alternative before reaching for a more complex algorithm.
A `std::vector` with O(n) insert is often faster than a `std::list` with O(1) insert because the linear scan is cache-friendly and the linked list traversal is not.
:::

## Trade-Off Summary

| Data structure | Best for | Avoid when |
|----------------|----------|-----------|
| `std::vector` | Most things; random access; sequential processing | Frequent front insertions |
| `std::list` | Frequent insert/delete *given a node pointer* | Sequential access; anything cache-sensitive |
| `std::deque` | Fast insert/delete at both ends | Random access in the middle |
| `std::stack` | LIFO processing | You need to access any element but the top |
| `std::queue` | FIFO processing | You need ordered access |
| `std::priority_queue` | Always need the min or max | You need arbitrary access or sorted output |
| `std::unordered_map` | Key-value lookups | Sorted iteration; adversarial keys |
| `std::map` | Sorted key-value; range queries | Highest-speed lookups |

## A Decision Flowchart

\index{data structure!decision flowchart}

When choosing a data structure, walk through these questions:

1. **Do you access elements by key (string, ID)?**
   Yes → hash table or BST.
   No → continue.

2. **Do you need random access by index?**
   Yes → `std::vector`.
   No → continue.

3. **Is LIFO ordering required?**
   Yes → `std::stack`.
   No → continue.

4. **Is FIFO ordering required?**
   Yes → `std::queue`.
   No → continue.

5. **Do you need the minimum or maximum repeatedly?**
   Yes → `std::priority_queue` (heap).
   No → continue.

6. **Do you frequently insert or delete at arbitrary positions?**
   Yes → `std::list` (if you have node pointers) or reconsider your design.
   No → `std::vector` is probably fine.

## Key Points

- No data structure is universally best --- choose based on your dominant operations.
- `std::vector` is the right default for most sequential data; its cache performance often outweighs its O(n) insert cost in practice.
- Hash tables provide average O(1) key-based access; trees provide O(log n) with sorted order.
- Heaps give O(1) min/max access and O(log n) insert/remove; everything else requires O(n) to find the extremum.
- Linked lists win only when you insert/delete at a *known position* and cache miss cost is acceptable.
- Always profile before replacing a simple structure with a complex one.

## Exercises

1. **Think about it:** You are building a leaderboard that stores player names and scores.
You need to: add or update a score in O(1), find the top-10 players in O(k log k), and look up any player's rank in O(log n).
Which combination of data structures would you use?

2. **What does this print?**

    ```cpp
    std::vector<int> v = {5, 3, 1, 4, 2};
    std::priority_queue<int, std::vector<int>, std::greater<int>> pq(v.begin(), v.end());

    while (!pq.empty()) {
        std::cout << pq.top() << " ";
        pq.pop();
    }
    std::cout << "\n";
    ```

3. **Calculation:** You have a `std::list<int>` with 1 million elements.
You iterate through the entire list 1000 times.
If each cache miss takes 100 ns and each cache hit takes 1 ns, and the list causes a cache miss every 8 elements (due to node layout), roughly how much time is spent on memory access for the entire operation?
Compare this to a `std::vector<int>` with one cache miss per 16 elements.

4. **Think about it:** You are implementing a spell-checker that needs to look up whether a word is in a dictionary of 100,000 words.
Lookups happen 10,000 times per document.
Which data structure would you use?
Would your answer change if you also need to suggest the closest match for misspelled words?

5. **Where is the bug in this design?**

    You are implementing a task scheduler.
    Tasks arrive continuously and must be processed in arrival order.
    You decide to use a `std::vector<Task>` and process from the front using `erase(begin())`.
    What is the problem with this design?
    What should you use instead?

6. **What does this print?**

    ```cpp
    std::map<std::string, int> m;
    m["Hobbit"] = 1937;
    m["Silmarillion"] = 1977;
    m["Unfinished Tales"] = 1980;

    for (const auto& [k, v] : m) {
        std::cout << k << ": " << v << "\n";
    }
    ```

7. **Calculation:** A program uses `std::list<int>` for a 10,000-element sequence and needs to access elements by index 500 times per second.
Each index access traverses from the head.
Assume the average access is at the middle (index 5,000), and each node visit takes 50 ns.
What is the total time per second spent on index accesses?
Would `std::vector` be faster?

8. **Think about it:** Why does `std::deque` exist if `std::vector` can be used for most things?
Give an example of a workload where `std::deque` is clearly better than `std::vector`.

9. **Think about it:** Hash tables have O(1) average but O(n) worst case.
BSTs (`std::map`) have O(log n) guaranteed.
In a real-time system where latency spikes are unacceptable, which would you prefer?
Why?

10. **Write a program** that benchmarks `std::vector` vs. `std::list` for sequential traversal of 100,000 integers.
Use `std::chrono::high_resolution_clock` to measure time.
Print the time for each and observe which is faster.
