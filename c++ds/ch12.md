# 12. Testing and Debugging Data Structures

\index{testing}
\index{debugging}

A data structure implementation that passes casual tests can still be wrong.
Edge cases, invalid inputs, and concurrent access reveal bugs that simple "it worked" tests miss.
This chapter covers systematic testing strategies for data structures: what to test, how to write tests that catch real bugs, how to use sanitizers, and how to debug pointer-based code when something goes wrong.

## What to Test

\index{testing!strategy}

Data structures have three categories of correctness requirements:

1. **Functional correctness:** operations return the right results.
2. **Invariant preservation:** internal state remains valid after every operation.
3. **Resource safety:** no memory leaks, no use-after-free, no double-free.

Test all three.
A test that only checks return values can miss corrupt internal state that causes a crash 100 operations later.

### Edge Cases for Every Data Structure

Every data structure has standard edge cases that must be tested:

| Case | Why it matters |
|------|---------------|
| Empty container | Most bugs involve null pointer dereference on empty |
| Single element | `head_` and `tail_` point to the same node |
| Two elements | Removing either element is different from removing a middle element |
| Full container (fixed-size) | Off-by-one in overflow detection |
| Transition through empty | After removing all elements, must be cleanly empty |
| Large input | Reveals O(n²) bugs and integer overflow in size counters |

## Writing Systematic Tests

\index{testing!systematic}

A good test function:

1. Constructs the data structure.
2. Calls an operation.
3. Asserts the expected result.
4. Checks invariants (if you have a checker).
5. Continues with the next operation.

```cpp
#include <cassert>
#include <iostream>
#include <stdexcept>
#include <string>

// Assume LinkedList<T> is defined (from Chapter 4)

void test_empty_list()
{
    LinkedList<std::string> list;
    assert(list.empty());
    assert(list.size() == 0);

    bool threw = false;
    try { list.remove_front(); }
    catch (const std::underflow_error&) { threw = true; }
    assert(threw);

    std::cout << "test_empty_list passed\n";
}

void test_single_element()
{
    LinkedList<std::string> list;
    list.append("Beloved");

    assert(!list.empty());
    assert(list.size() == 1);
    assert(list.front() == "Beloved");

    list.remove_front();
    assert(list.empty());
    assert(list.size() == 0);

    std::cout << "test_single_element passed\n";
}

void test_multiple_elements()
{
    LinkedList<std::string> list;
    list.append("Beloved");
    list.append("Sula");
    list.append("Jazz");

    assert(list.size() == 3);
    assert(list.front() == "Beloved");

    list.remove_front();
    assert(list.size() == 2);
    assert(list.front() == "Sula");

    list.remove_front();
    list.remove_front();
    assert(list.empty());

    std::cout << "test_multiple_elements passed\n";
}

void test_prepend()
{
    LinkedList<std::string> list;
    list.prepend("Jazz");
    list.prepend("Sula");
    list.prepend("Beloved");

    assert(list.front() == "Beloved");
    assert(list.size() == 3);

    std::cout << "test_prepend passed\n";
}

int main()
{
    test_empty_list();
    test_single_element();
    test_multiple_elements();
    test_prepend();

    std::cout << "All tests passed.\n";
    return 0;
}
```

```
test_empty_list passed
test_single_element passed
test_multiple_elements passed
test_prepend passed
All tests passed.
```

## Testing with Sanitizers

\index{sanitizer!testing}

The best tests do not just check correctness --- they check for memory safety.
Compile with sanitizers enabled during testing:

```bash
g++ -std=c++23 -Wall -Wextra -pedantic -fsanitize=address,undefined -g test.cpp -o test
./test
```

**AddressSanitizer (ASan)** catches:
- Buffer overflows (reading or writing past the end of an array)
- Use-after-free (accessing a node after it has been deleted)
- Double-free (calling `delete` twice on the same pointer)
- Memory leaks (allocated but never freed)

**UndefinedBehaviorSanitizer (UBSan)** catches:
- Signed integer overflow
- Null pointer dereference
- Out-of-bounds array access (for stack arrays)

```cpp
#include <iostream>

int main()
{
    int arr[5] = {1, 2, 3, 4, 5};
    // This buffer overflow will be caught by ASan
    // std::cout << arr[10] << "\n";

    // This use-after-free will be caught by ASan
    int* p = new int(42);
    delete p;
    // std::cout << *p << "\n";  // use-after-free

    std::cout << "Clean run with sanitizers\n";
    return 0;
}
```

```
Clean run with sanitizers
```

::: {.tip}
**Tip:** Run every test with ASan and UBSan enabled.
Many data structure bugs --- especially in linked list and heap implementations --- produce no visible error in a normal run but are caught immediately by sanitizers.
:::

## Testing Copy and Move Operations

\index{testing!copy and move}

If your data structure implements the Big Five, test them explicitly:

```cpp
void test_copy()
{
    LinkedList<std::string> original;
    original.append("Beloved");
    original.append("Sula");

    LinkedList<std::string> copy = original;  // copy constructor

    // Modifying copy must not affect original
    copy.append("Jazz");
    assert(original.size() == 2);
    assert(copy.size() == 3);

    std::cout << "test_copy passed\n";
}

void test_move()
{
    LinkedList<std::string> source;
    source.append("Beloved");
    source.append("Sula");

    LinkedList<std::string> moved = std::move(source);  // move constructor

    assert(moved.size() == 2);
    assert(moved.front() == "Beloved");
    // source is in valid but unspecified state after move
    assert(source.empty() || source.size() >= 0);  // just must not crash

    std::cout << "test_move passed\n";
}
```

::: {.tip}
**Trap:** It is easy to accidentally test the same code path twice and think you have more coverage than you do.
After `b = a` (copy), modify `b` and check that `a` is unchanged.
This is the test that actually catches shallow-copy bugs.
:::

## Debugging Pointer-Based Code

\index{debugging!pointer-based}

When your linked data structure crashes or produces wrong output:

### Strategy 1: Add an Invariant Checker

Write a `check_invariants()` function (see Chapter 10) and call it at the start and end of every method.
The checker will identify *where* the invariant was first violated.

### Strategy 2: Print the Internal State

Add a `dump()` method that prints every node:

```cpp
void dump() const
{
    std::cout << "size=" << size_ << " head=" << head_.get()
              << " tail=" << tail_ << "\n";
    int i = 0;
    for (const Node* p = head_.get(); p; p = p->next.get()) {
        std::cout << "  [" << i++ << "] " << p->value
                  << " prev=" << p->prev
                  << " next=" << p->next.get() << "\n";
    }
}
```

Printing raw pointers alongside values lets you verify that `prev` and `next` links are consistent.

### Strategy 3: Use a Debugger

Compile with `-g -O0` and run under `gdb` or `lldb`:

```bash
g++ -std=c++23 -g -O0 -fsanitize=address test.cpp -o test
gdb ./test
(gdb) run
(gdb) backtrace   # see the call stack at the crash
(gdb) info locals # see local variables at the crash site
```

### Strategy 4: Bisect

If a test fails after many operations, binary-search for the first failing operation.
Add a counter and call `check_invariants()` every N operations, halving N until you find the exact operation that corrupts the structure.

## Property-Based Testing

\index{property-based testing}

Instead of writing individual tests, **property-based testing** generates random inputs and checks that properties hold for all of them.

For a stack, the property is:
> For any sequence of pushes, `size()` equals the number of items pushed, and `pop()` returns them in LIFO order.

```cpp
#include <cassert>
#include <cstdlib>
#include <iostream>
#include <stack>
#include <string>
#include <vector>

void fuzz_stack(int iterations)
{
    // Use std::stack as a reference implementation
    std::stack<int> reference;
    Stack<int> ours;  // your implementation

    for (int i = 0; i < iterations; ++i) {
        int op = std::rand() % 3;

        if (op == 0 || reference.empty()) {
            // Push a random value
            int v = std::rand() % 1000;
            reference.push(v);
            ours.push(v);
        } else if (op == 1) {
            // Pop and compare
            assert(reference.top() == ours.top());
            reference.pop();
            ours.pop();
        }

        // Check size invariant
        assert(static_cast<int>(reference.size()) == ours.size());
    }

    std::cout << "Fuzz test passed (" << iterations << " iterations)\n";
}

int main()
{
    std::srand(42);
    fuzz_stack(10000);
    return 0;
}
```

This approach catches bugs that carefully hand-written test cases miss, because it exercises sequences of operations you would never think to test manually.

Write a test suite for your `Stack<T>` template that covers:
- Empty stack operations (should throw)
- Push and pop in LIFO order
- Size tracking through multiple operations
- Copy constructor and copy assignment
- Move constructor and move assignment
- Fuzz test against `std::stack` as reference

## Key Points

- Test **functional correctness**, **invariant preservation**, and **resource safety** --- not just return values.
- Always test: empty, single element, two elements, full, and the transition back through empty.
- Run tests with `fsanitize=address,undefined` to catch memory bugs silently lurking in the code.
- Test copy and move operations by checking that the copies are truly independent.
- Add a `dump()` method and an invariant checker for debug builds.
- **Property-based (fuzz) testing** generates random operation sequences and checks invariants after each.
  It finds bugs that hand-written tests miss.
- Use `std::stack`, `std::queue`, or other standard containers as **reference implementations** to validate your own.
