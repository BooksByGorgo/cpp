# 10. Design and Implementation Best Practices

\index{design!best practices}

Writing a data structure that works is one thing.
Writing one that is correct under all conditions, easy to test, maintainable, and efficient is another.
This chapter covers the practices that separate production-quality data structure implementations from prototype code: invariant-driven design, separation of interface from implementation, exception safety, iterator design, and the principle of least surprise.

## Invariant-Driven Design

\index{invariant!design}

Every data structure has **invariants** --- conditions that must be true at all times.
Identifying and enforcing your invariants is the single most effective technique for writing correct code.

For a doubly linked list, the invariants include:

- `head_->prev == nullptr`
- `tail_->next == nullptr`
- Every node's `next->prev == node`
- `size_` equals the number of nodes in the chain
- If `size_ == 0`, then `head_ == nullptr` and `tail_ == nullptr`

Document your invariants in comments.
At the start of every public method, the invariants hold.
At the end, they must hold again.

```cpp
// In debug builds, check the invariants explicitly
#ifdef NDEBUG
    #define CHECK_INVARIANTS() ((void)0)
#else
    #define CHECK_INVARIANTS() check_invariants()
#endif

void DoublyLinkedList::check_invariants() const
{
    if (size_ == 0) {
        assert(head_ == nullptr && tail_ == nullptr);
        return;
    }
    assert(head_ != nullptr && tail_ != nullptr);
    assert(head_->prev == nullptr);
    assert(tail_->next == nullptr);

    int count = 0;
    const Node* p = head_.get();
    while (p->next) {
        assert(p->next->prev == p);
        p = p->next.get();
        ++count;
    }
    assert(p == tail_);
    assert(count + 1 == size_);
}
```

Invariant checks are invaluable during development.
Use `assert` so they compile away in release builds with `-DNDEBUG`.

::: {.tip}
**Tip:** Write your invariant checker before writing any other code.
It forces you to think precisely about what your data structure must guarantee, and it catches bugs immediately when you violate them during development.
:::

## Separating Interface from Implementation

\index{interface!separation}

Users of your data structure should only see the operations they need --- not the internal representation.
Keep all internal fields `private`, expose only the public interface, and use accessors rather than exposing raw members.

```cpp
// Bad: exposes internals
class BadStack {
public:
    std::string* data_;   // public!
    int top_;             // public!
    void push(const std::string& v) { data_[++top_] = v; }
};

// Good: hides internals
class GoodStack {
public:
    void push(const std::string& v);
    std::string pop();
    const std::string& top() const;
    bool empty() const;
    int size() const;
private:
    std::string data_[64];
    int top_ = -1;
};
```

Clients that use `BadStack::data_` directly will break the moment you change the internal representation.
`GoodStack` can change its internals (switch to `std::vector`, add bounds checking) without breaking any caller.

## Exception Safety

\index{exception safety}

Exception safety levels:

- **No-throw guarantee:** the function never throws.
  Mark with `noexcept`.
- **Strong guarantee:** if a function throws, the object is left in its original state (as if the call never happened).
- **Basic guarantee:** if a function throws, the object is in a valid (but possibly changed) state.
  No resources are leaked.
- **No guarantee:** the object may be in an invalid state after a throw.

Most operations should provide at least the basic guarantee.
The **copy-and-swap idiom** makes copy assignment strongly exception-safe:

```cpp
#include <algorithm>
#include <string>

class MyList {
public:
    // Copy-and-swap: strongly exception safe
    MyList& operator=(MyList other)   // by value: copy already made
    {
        std::swap(data_, other.data_);
        std::swap(size_, other.size_);
        std::swap(capacity_, other.capacity_);
        return *this;
    }
    // ... rest of the class
private:
    std::string* data_ = nullptr;
    int size_ = 0;
    int capacity_ = 0;
};
```

If the copy constructor throws, the original object is untouched because the copy is made in the parameter.
The swap itself is `noexcept`.

::: {.tip}
**Tip:** Mark `swap`, move constructor, and move assignment `noexcept`.
The standard library can make significant optimizations (like avoiding copies in `std::vector` reallocation) when move operations are `noexcept`.
:::

## Iterator Design

\index{iterator}

Iterators give users a way to traverse your data structure without exposing its internals.
A compliant iterator supports:

- `operator*()`: dereference to get the current element.
- `operator++()`: advance to the next element.
- `operator!=()`: check if two iterators are at different positions.

These three are enough for range-for loops:

```cpp
#include <iostream>
#include <string>

// Minimal forward iterator for a linked list
template<typename T>
class LinkedList {
public:
    struct Node {
        T value;
        Node* next = nullptr;
        explicit Node(const T& v) : value{v} {}
    };

    struct Iterator {
        Node* node;
        explicit Iterator(Node* n) : node{n} {}

        T& operator*() { return node->value; }
        const T& operator*() const { return node->value; }
        Iterator& operator++() { node = node->next; return *this; }
        bool operator!=(const Iterator& other) const { return node != other.node; }
    };

    Iterator begin() { return Iterator{head_}; }
    Iterator end() { return Iterator{nullptr}; }

    void append(const T& v)
    {
        Node* node = new Node{v};
        if (!tail_) { head_ = tail_ = node; }
        else { tail_->next = node; tail_ = node; }
        ++size_;
    }

    ~LinkedList()
    {
        while (head_) {
            Node* tmp = head_->next;
            delete head_;
            head_ = tmp;
        }
    }

private:
    Node* head_ = nullptr;
    Node* tail_ = nullptr;
    int size_ = 0;
};

int main()
{
    LinkedList<std::string> list;
    list.append("Ulysses");
    list.append("Dubliners");
    list.append("Portrait");

    for (const auto& v : list) {
        std::cout << v << "\n";
    }

    return 0;
}
```

```
Ulysses
Dubliners
Portrait
```

::: {.tip}
**Trap:** Modifying a container while iterating over it (adding or removing elements) usually invalidates iterators.
This is called **iterator invalidation**.
`std::vector` invalidates all iterators on reallocation; `std::list` only invalidates the iterator pointing to the erased element.
Document which operations invalidate your iterators.
:::

## The Principle of Least Surprise

\index{principle of least surprise}

Your data structure should behave the way users expect.
Follow the conventions established by the standard library:

- `size()` returns the number of elements.
- `empty()` returns `true` if there are no elements.
- `clear()` removes all elements.
- Copying the container copies all elements.
- Move constructor/assignment should be `noexcept`.

Deviating from these conventions creates bugs in code that uses your structure.

### Value Semantics

Data structures should support **value semantics**: copies are independent, and the original is not affected by changes to a copy.
This matches how built-in types work and how the standard library works.

```cpp
// Good: copying a vector gives an independent copy
std::vector<int> a = {1, 2, 3};
std::vector<int> b = a;
b.push_back(4);
// a still has 3 elements; b has 4
```

If your data structure uses shared state between copies (pointer aliasing without deep copy), it violates value semantics and will surprise callers.

## Key Points

- **Invariants** describe what must always be true about your data structure.
  Document them and check them in debug builds with `assert`.
- **Separate interface from implementation**: keep all internal fields `private`.
  Clients depend on the interface, not the internals.
- Aim for at least the **basic exception safety guarantee** in all operations.
  Use copy-and-swap for strongly exception-safe copy assignment.
- Mark `swap`, move constructor, and move assignment `noexcept` to enable standard library optimizations.
- **Iterators** give clients access to elements without exposing internals.
  Implement `operator*`, `operator++`, and `operator!=` for range-for support.
- **Iterator invalidation** rules must be documented and consistent.
- Follow **principle of least surprise**: match standard library conventions for naming and behavior.
- **Value semantics**: copies are independent; modifying one does not affect the other.
