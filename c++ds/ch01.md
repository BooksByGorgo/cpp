# 1. Abstract Data Types

\index{abstract data type}

You have already written classes that bundle data and operations together.
But there is a deeper idea hiding in that pattern: the difference between *what* a data structure does and *how* it does it.
A **abstract data type** (ADT) is a mathematical model of a data structure defined entirely by its operations and their behavior --- with no mention of implementation.
Understanding ADTs gives you a vocabulary for talking about data structures precisely, and it helps you design code that is not tightly coupled to any particular implementation.
In this chapter you will learn what ADTs are, how they differ from concrete data structures, how to express them in C++, and why the implementation choices you make later actually matter.

## What Is an Abstract Data Type?

\index{abstract data type!definition}

An **abstract data type** describes a collection of values and a set of operations on those values, defined by what the operations *do* --- their preconditions, postconditions, and invariants --- rather than by how they are coded.

Think about a stack of books.
You can describe it completely without ever mentioning memory layout:

- **push(item):** Place an item on top.
- **pop():** Remove and return the item on top.
  Precondition: the stack is not empty.
- **top():** Return (without removing) the item on top.
  Precondition: the stack is not empty.
- **empty():** Return true if there are no items.
- **size():** Return the number of items.

That description is the Stack ADT.
It says nothing about arrays, linked lists, or any other mechanism.
It is a contract.

### ADTs vs. Data Structures

\index{abstract data type!vs. data structure}

An **ADT** is the contract; a **data structure** is a specific implementation of that contract.

| Concept | Stack ADT | Array-based stack | Linked-list stack |
|---------|-----------|------------------|------------------|
| What it is | A specification | A concrete class | A concrete class |
| Defines behavior | Yes | Yes (fulfills ADT) | Yes (fulfills ADT) |
| Specifies memory | No | Yes (contiguous array) | Yes (heap nodes) |

You can have many different data structures that implement the same ADT.
As long as they fulfill the contract, they are interchangeable from the user's perspective.

::: {.tip}
**Tip:** Always think about the ADT first --- what operations do you need and what do they guarantee? --- before choosing an implementation.
The implementation decision is separate from the design decision.
:::

## Interfaces, Operations, and Encapsulation

\index{interface}
\index{encapsulation}

In C++ you express an ADT as an abstract class: a class with pure virtual functions that define the operations.
The abstract class is the interface; concrete classes are the implementations.

```cpp
#include <string>

// The Stack ADT: defines the contract, not the implementation
class Stack {
public:
    virtual ~Stack() = default;

    // Add item to the top
    virtual void push(const std::string& item) = 0;

    // Remove and return the top item. Precondition: !empty()
    virtual std::string pop() = 0;

    // Return (without removing) the top item. Precondition: !empty()
    virtual const std::string& top() const = 0;

    // True if there are no items
    virtual bool empty() const = 0;

    // Number of items
    virtual int size() const = 0;
};
```

The `Stack` class cannot be instantiated --- it is an interface.
Any class that overrides all five pure virtual functions is a valid Stack implementation.

Here is a simple array-based implementation that fulfills the contract:

```cpp
#include <iostream>
#include <stdexcept>
#include <string>

class Stack {
public:
    virtual ~Stack() = default;
    virtual void push(const std::string& item) = 0;
    virtual std::string pop() = 0;
    virtual const std::string& top() const = 0;
    virtual bool empty() const = 0;
    virtual int size() const = 0;
};

class ArrayStack : public Stack {
public:
    ArrayStack() : top_{-1} {}

    void push(const std::string& item) override
    {
        if (top_ >= kCapacity - 1) {
            throw std::overflow_error("stack is full");
        }
        data_[++top_] = item;
    }

    std::string pop() override
    {
        if (empty()) {
            throw std::underflow_error("stack is empty");
        }
        return data_[top_--];
    }

    const std::string& top() const override
    {
        if (empty()) {
            throw std::underflow_error("stack is empty");
        }
        return data_[top_];
    }

    bool empty() const override { return top_ == -1; }
    int size() const override { return top_ + 1; }

private:
    static constexpr int kCapacity = 100;
    std::string data_[kCapacity];
    int top_;
};

int main()
{
    ArrayStack s;
    s.push("Dune");
    s.push("Foundation");
    s.push("Neuromancer");

    std::cout << "Top: " << s.top() << "\n";
    std::cout << "Size: " << s.size() << "\n";

    while (!s.empty()) {
        std::cout << s.pop() << "\n";
    }

    return 0;
}
```

```
Top: Neuromancer
Size: 3
Neuromancer
Foundation
Dune
```

The `main` function only uses the operations defined by the Stack ADT.
It does not know --- or care --- that the implementation uses an array internally.
If you replaced `ArrayStack` with a linked-list implementation, `main` would not need to change.

### Preconditions and Invariants

\index{precondition}
\index{invariant}

An ADT's contract includes more than just the operation names.
It includes **preconditions** (what must be true before calling an operation), **postconditions** (what is guaranteed to be true after), and **invariants** (what is always true about the data structure).

For the Stack ADT:

- **Precondition** of `pop()`: `!empty()`.
  Calling `pop()` on an empty stack is undefined behavior per the contract.
  The implementation may throw an exception or crash --- either is valid.
- **Invariant**: `size() >= 0` is always true.
- **Postcondition** of `push(item)`: `!empty()` and `top() == item`.

When you implement an ADT, enforcing preconditions and maintaining invariants is your responsibility.

::: {.tip}
**Trap:** Ignoring preconditions is a common source of hard-to-find bugs.
If your code calls `pop()` on a potentially empty stack without checking `empty()` first, you have violated the contract.
:::

## Why Implementation Choices Matter

\index{implementation!choices}

The ADT defines *what* you need.
The implementation defines *how* you get it.
Different implementations of the same ADT can have wildly different performance characteristics.

Consider a Queue ADT with `enqueue`, `dequeue`, `front`, `empty`, and `size` operations.
You could implement it with:

- An array where you shift elements on every `dequeue` --- `enqueue` is O(1), `dequeue` is O(n).
- A circular array --- both `enqueue` and `dequeue` are O(1), but capacity is fixed.
- A linked list --- both are O(1) and capacity is unlimited, but each node allocates heap memory.

All three are valid Queue implementations.
They fulfill the same contract.
But if you are building a real-time system that dequeues millions of items per second, the array-shift implementation will be painfully slow and the circular array or linked list will be fine.
If you are building something with unpredictable queue sizes, the fixed-capacity circular array will cause overflow problems.

The choice depends on your workload.
That is why this book spends time analyzing the performance of each implementation --- so you can choose the right one.

::: {.tip}
**Tip:** The ADT tells you the interface.
The performance analysis tells you which implementation to choose.
Study both.
:::

### The Cost of Abstraction

\index{abstraction!cost}

Pure abstract class interfaces (with virtual dispatch) have a small runtime cost: each virtual call goes through a vtable pointer.
For most applications this is negligible.
But for data structures that are called millions of times in tight loops, the overhead adds up.

C++ templates (Chapter 11) offer a zero-overhead alternative: the interface is enforced at compile time, not run time.
The standard library containers (`std::vector`, `std::list`, `std::stack`) use templates precisely because they need to avoid virtual dispatch overhead.

The tradeoff is flexibility: virtual interfaces let you swap implementations at runtime; templates do not.
Understanding this tradeoff will shape how you design your own data structures.

## Try It: Defining and Implementing an ADT

Here is a starter program to experiment with ADTs.
Define your own simple ADT, implement it two different ways, and verify they behave identically:

```cpp
#include <iostream>
#include <stdexcept>
#include <string>

// Bag ADT: unordered collection that supports insert, remove, contains, size
class Bag {
public:
    virtual ~Bag() = default;
    virtual void insert(const std::string& item) = 0;
    virtual void remove(const std::string& item) = 0;
    virtual bool contains(const std::string& item) const = 0;
    virtual int size() const = 0;
    virtual bool empty() const = 0;
};

// Simple array-based implementation
class ArrayBag : public Bag {
public:
    ArrayBag() : count_{0} {}

    void insert(const std::string& item) override
    {
        if (count_ >= kCapacity) {
            throw std::overflow_error("bag is full");
        }
        data_[count_++] = item;
    }

    void remove(const std::string& item) override
    {
        for (int i = 0; i < count_; ++i) {
            if (data_[i] == item) {
                data_[i] = data_[--count_];
                return;
            }
        }
        throw std::invalid_argument("item not found");
    }

    bool contains(const std::string& item) const override
    {
        for (int i = 0; i < count_; ++i) {
            if (data_[i] == item) return true;
        }
        return false;
    }

    int size() const override { return count_; }
    bool empty() const override { return count_ == 0; }

private:
    static constexpr int kCapacity = 50;
    std::string data_[kCapacity];
    int count_;
};

int main()
{
    ArrayBag bag;
    bag.insert("Dune");
    bag.insert("Foundation");
    bag.insert("Neuromancer");

    std::cout << "Size: " << bag.size() << "\n";
    std::cout << "Contains Dune: " << bag.contains("Dune") << "\n";

    bag.remove("Foundation");
    std::cout << "After remove, size: " << bag.size() << "\n";
    std::cout << "Contains Foundation: " << bag.contains("Foundation") << "\n";

    return 0;
}
```

```
Size: 3
Contains Dune: 1
After remove, size: 2
Contains Foundation: 0
```

Try adding a second implementation (perhaps one that keeps items sorted) and verify it passes the same tests.

## Key Points

- An **abstract data type** defines a collection and its operations by their *behavior*, not their implementation.
- The ADT is a contract: it specifies preconditions, postconditions, and invariants.
- A **data structure** is a concrete implementation of an ADT.
- Multiple data structures can fulfill the same ADT with different performance characteristics.
- In C++, an ADT is expressed as an abstract class with pure virtual functions.
- Encapsulation hides the implementation from users of the ADT, letting you swap implementations without changing calling code.
- Different implementations of the same ADT can have very different time and space complexity --- choosing the right one depends on your workload.
- Virtual dispatch has a small runtime cost; templates offer a zero-overhead compile-time alternative.
