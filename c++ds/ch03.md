# 3. Linear Data Structures Overview

\index{linear data structure}

A **linear data structure** arranges its elements in a sequence, one after another.
Each element has at most one predecessor and one successor (except at the ends).
Arrays, linked lists, stacks, and queues are all linear.
Before studying each one in detail, this chapter surveys the common operations they all share --- traversal, insertion, and deletion --- and compares the two fundamental implementation strategies: array-based and pointer-based.
Understanding these trade-offs at a high level will help you make good choices throughout the rest of the book.

## Traversal, Insertion, and Deletion

\index{traversal}
\index{insertion}
\index{deletion}

Every linear data structure supports some form of these three operations:

- **Traversal:** Visit every element, usually in sequence.
- **Insertion:** Add a new element at some position.
- **Deletion:** Remove an element at some position.

The cost of each operation depends heavily on *where* it happens and on the underlying representation.

### Traversal

Traversal visits each element once.
For any linear structure, traversal is O(n) --- you must touch every element.
The constant factor depends on memory layout: arrays have excellent cache locality because elements are contiguous in memory; linked lists have poor cache locality because nodes are scattered across the heap.

```cpp
#include <iostream>
#include <string>

// Array traversal
void print_array(const std::string* arr, int n)
{
    for (int i = 0; i < n; ++i) {
        std::cout << arr[i] << "\n";
    }
}

int main()
{
    std::string books[] = {"Hamlet", "Othello", "Macbeth"};
    print_array(books, 3);
    return 0;
}
```

```
Hamlet
Othello
Macbeth
```

### Insertion

Where you insert determines the cost:

| Insert position | Array | Linked list |
|----------------|-------|-------------|
| At the end | O(1) amortized | O(1) with tail pointer |
| At the front | O(n) (must shift) | O(1) |
| At arbitrary index | O(n) (must shift) | O(n) to find position, O(1) to insert |

```cpp
#include <iostream>
#include <string>

// Insert at position pos in an array (shifting elements right)
void insert_at(std::string* arr, int& size, int pos, const std::string& value)
{
    // Shift elements right to make room
    for (int i = size; i > pos; --i) {
        arr[i] = arr[i - 1];
    }
    arr[pos] = value;
    ++size;
}

int main()
{
    std::string books[5] = {"Hamlet", "Macbeth"};
    int size = 2;

    insert_at(books, size, 1, "Othello");  // insert between Hamlet and Macbeth

    for (int i = 0; i < size; ++i) {
        std::cout << books[i] << "\n";
    }

    return 0;
}
```

```
Hamlet
Othello
Macbeth
```

The shift loop makes this O(n) in the worst case --- inserting at index 0 moves every element.

### Deletion

Deletion mirrors insertion:

| Delete position | Array | Linked list |
|----------------|-------|-------------|
| At the end | O(1) | O(1) with tail pointer |
| At the front | O(n) (must shift) | O(1) |
| At arbitrary index | O(n) (must shift) | O(n) to find, O(1) to delete |

## Array-Based vs. Pointer-Based Implementations

\index{array-based implementation}
\index{pointer-based implementation}

The two fundamental implementation strategies for linear structures are array-based (contiguous memory) and pointer-based (nodes linked by pointers).
Each has strengths and weaknesses.

### Array-Based

An array-based implementation stores elements in a contiguous block of memory.

**Strengths:**
- O(1) random access by index.
- Excellent cache performance: elements are adjacent in memory.
- Simple implementation.
- Low memory overhead: no per-element pointers.

**Weaknesses:**
- Fixed capacity (unless you implement a growable buffer with copying).
- O(n) insertion and deletion in the middle.
- May waste space if capacity is much larger than actual size.

```cpp
#include <iostream>
#include <string>

class ArrayList {
public:
    ArrayList() : size_{0} {}

    void append(const std::string& v) { data_[size_++] = v; }

    // O(1) --- direct index
    const std::string& at(int i) const { return data_[i]; }

    int size() const { return size_; }

private:
    static constexpr int kCap = 64;
    std::string data_[kCap];
    int size_;
};

int main()
{
    ArrayList list;
    list.append("Hamlet");
    list.append("Othello");
    list.append("Macbeth");

    // Random access in O(1)
    std::cout << list.at(1) << "\n";

    return 0;
}
```

```
Othello
```

### Pointer-Based (Linked)

A pointer-based implementation stores each element in a **node** that contains the element and a pointer (or pointers) to neighboring nodes.

**Strengths:**
- O(1) insertion and deletion at the front (or at any position given a pointer to the predecessor).
- Dynamic size: nodes are allocated on demand, no capacity limit.
- No wasted space for unused slots.

**Weaknesses:**
- O(n) to find element by index (must traverse).
- Poor cache performance: nodes are scattered in heap memory.
- Per-element pointer overhead (8 bytes on 64-bit systems).

```cpp
#include <iostream>
#include <memory>
#include <string>

struct Node {
    std::string value;
    std::unique_ptr<Node> next;

    explicit Node(const std::string& v) : value{v} {}
};

class LinkedList {
public:
    LinkedList() : head_{nullptr}, size_{0} {}

    // O(1) prepend
    void prepend(const std::string& v)
    {
        auto node = std::make_unique<Node>(v);
        node->next = std::move(head_);
        head_ = std::move(node);
        ++size_;
    }

    int size() const { return size_; }

    void print() const
    {
        for (const Node* p = head_.get(); p; p = p->next.get()) {
            std::cout << p->value << "\n";
        }
    }

private:
    std::unique_ptr<Node> head_;
    int size_;
};

int main()
{
    LinkedList list;
    list.prepend("Macbeth");
    list.prepend("Othello");
    list.prepend("Hamlet");

    list.print();

    return 0;
}
```

```
Hamlet
Othello
Macbeth
```

## Choosing the Right Representation

\index{data structure!selection}

The right choice depends on your workload.
Ask yourself:

1. **Do you need random access by index?**
   If yes, use an array.
   Linked lists require O(n) traversal to reach element *i*.

2. **Do you insert or delete frequently at the front or middle?**
   If yes, a linked list avoids the O(n) shift cost.

3. **Do you know the maximum size in advance?**
   If yes, a fixed-size array is simple and fast.
   If no, you need either a growable array or a linked list.

4. **Is cache performance critical?**
   If yes, arrays win every time.
   Linked list traversal generates many cache misses.

5. **Is memory overhead a concern?**
   Each linked list node carries a pointer (8 bytes on 64-bit).
   For small elements (e.g., `int`, 4 bytes), the overhead is 200%.
   For large elements, it matters less.

### The Standard Library

The C++ standard library provides well-tested implementations:

| Type | Standard class | Implementation |
|------|---------------|---------------|
| Growable array | `std::vector<T>` | Dynamic array |
| Doubly linked list | `std::list<T>` | Pointer-based |
| Singly linked list | `std::forward_list<T>` | Pointer-based |
| Double-ended queue | `std::deque<T>` | Chunked array |

::: {.tip}
**Tip:** In production code, prefer the standard library containers over hand-rolled ones.
They are heavily tested, handle edge cases correctly, and are optimized for most workloads.
Implement your own only to learn the internals or when profiling shows the standard library is not fast enough.
:::

## Lists

\index{list}

A **list** is a linear sequence of elements that supports:

- Access by position
- Insertion at any position
- Deletion at any position
- Traversal

Lists are the most general linear structure --- stacks and queues are specialized lists with restricted access.
Chapter 4 covers the List ADT and its implementations in detail.

::: {.tip}
**Wut:** `std::list<T>` is a doubly linked list, not a growable array.
If you want "a list that grows dynamically", you almost certainly want `std::vector<T>`.
`std::list` is only better than `std::vector` in the rare case where you frequently insert or delete in the middle and have an iterator to the position.
:::

## Try It: Comparing Array and Linked Representations

Here is a simple program that builds the same sequence both ways and compares access patterns.
Type it in and experiment:

```cpp
#include <iostream>
#include <memory>
#include <string>

// Array-based sequence
struct ArraySeq {
    std::string data[8];
    int size = 0;

    void append(const std::string& v) { data[size++] = v; }
    const std::string& at(int i) const { return data[i]; }
};

// Linked sequence
struct LNode {
    std::string value;
    std::unique_ptr<LNode> next;
    explicit LNode(const std::string& v) : value{v} {}
};

struct LinkedSeq {
    std::unique_ptr<LNode> head;
    int size = 0;

    void append(const std::string& v)
    {
        auto node = std::make_unique<LNode>(v);
        if (!head) {
            head = std::move(node);
        } else {
            LNode* p = head.get();
            while (p->next) p = p->next.get();
            p->next = std::move(node);
        }
        ++size;
    }

    // O(n) index access
    const std::string& at(int i) const
    {
        const LNode* p = head.get();
        for (int j = 0; j < i; ++j) p = p->next.get();
        return p->value;
    }
};

int main()
{
    ArraySeq arr;
    LinkedSeq lnk;

    for (const std::string& t : {"Hamlet", "Othello", "Macbeth"}) {
        arr.append(t);
        lnk.append(t);
    }

    std::cout << "Array [1]: " << arr.at(1) << "\n";
    std::cout << "Linked [1]: " << lnk.at(1) << "\n";

    return 0;
}
```

```
Array [1]: Othello
Linked [1]: Othello
```

Try timing the `at()` call for large sequences to observe the O(1) vs. O(n) difference.

## Key Points

- **Linear data structures** arrange elements in a sequence with O(n) traversal.
- **Traversal** is always O(n) regardless of representation.
- **Insertion and deletion** cost depends on position and representation.
- **Array-based:** O(1) random access, poor at front/middle insert/delete, excellent cache locality.
- **Pointer-based:** O(1) front insert/delete, O(n) index access, poor cache locality.
- Choose based on your dominant operation: index access favors arrays; front/middle insert favors linked lists.
- For cache-sensitive code, arrays almost always outperform linked lists despite the theoretical O(1) linked list advantages.
- In production, prefer `std::vector`, `std::list`, or `std::deque` over hand-rolled implementations.
