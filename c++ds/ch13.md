# 13. Documentation and Interface Design

\index{documentation}
\index{interface design}

Code is read far more often than it is written.
A data structure with a clean interface and good documentation is easier to use correctly, harder to misuse, and cheaper to maintain.
This chapter covers how to design interfaces that are hard to use wrong, how to document them effectively in C++ (Doxygen comments, preconditions, postconditions, thread-safety notes), and how to version a public API without breaking existing users.

## Designing Interfaces That Are Hard to Misuse

\index{interface!usability}

The best interface design makes the right way to use the code the easy way, and makes the wrong way a compile error.

### Express Invariants in the Type System

Instead of documenting that an index must be non-negative, use an unsigned type or a wrapper:

```cpp
// Bad: any int can be passed; negative values are a silent bug
std::string at(int index) const;

// Better: signed with a runtime check
std::string at(int index) const
{
    if (index < 0 || index >= size_) throw std::out_of_range("invalid index");
    return data_[index];
}

// Best for public APIs: use a named type
using Index = std::size_t;
std::string at(Index index) const;
```

### Use `[[nodiscard]]` for Operations That Return Values

\index{nodiscard}

If the caller ignores the return value of `pop()`, they silently discard data.
`[[nodiscard]]` turns this into a compiler warning:

```cpp
[[nodiscard]] std::string pop();
[[nodiscard]] bool empty() const;
```

Now `stack.pop();` (without using the return value) generates a warning.

### Separate Query from Mutation

Operations that only read state should be `const`.
Operations that modify state should not.
This lets callers reason about thread safety and prevents accidental mutations:

```cpp
// Query (const): does not modify the stack
const T& top() const;
bool empty() const;
int size() const;

// Mutation (non-const): modifies the stack
void push(const T& item);
T pop();
```

### Avoid Implicit Conversions

\index{explicit constructor}

Implicit conversions let the compiler silently convert a type into your type, hiding bugs:

```cpp
// Bad: int silently converts to Stack
class Stack {
public:
    Stack(int capacity) : capacity_{capacity} {}
    // ...
};

void process(Stack& s);
process(42);  // compiles! Creates a Stack with capacity 42 --- probably not intended

// Good: explicit prevents the silent conversion
class Stack {
public:
    explicit Stack(int capacity) : capacity_{capacity} {}
    // ...
};

process(42);  // error: no viable conversion from int to Stack
```

::: {.tip}
**Tip:** Make single-argument constructors `explicit` unless implicit conversion is genuinely useful.
Almost every "magic number passed as an object" bug is caused by a non-explicit constructor.
:::

## Writing Effective Documentation

\index{documentation!Doxygen}

The C++ community uses **Doxygen** for API documentation.
Doxygen parses specially formatted comments and generates HTML, LaTeX, or PDF reference docs.

A well-documented class looks like this:

```cpp
/**
 * @brief A max-heap priority queue.
 *
 * Stores elements and always provides O(1) access to the maximum element.
 * Insertion and removal are O(log n).
 *
 * @tparam T The element type. Must support operator< and operator>.
 *
 * @invariant data_[0] is the maximum element if the heap is not empty.
 * @invariant For every index i, data_[i] >= data_[2*i+1] and data_[2*i+2]
 *            (heap order property).
 */
template<typename T>
class MaxHeap {
public:
    /**
     * @brief Insert an element into the heap.
     * @param item The element to insert.
     * @post size() is one greater than before the call.
     * @post max() >= item.
     */
    void insert(const T& item);

    /**
     * @brief Remove and return the maximum element.
     * @return The maximum element.
     * @pre !empty()
     * @throws std::underflow_error if the heap is empty.
     * @post size() is one less than before the call.
     */
    [[nodiscard]] T remove_max();

    /**
     * @brief Return the maximum element without removing it.
     * @return A const reference to the maximum element.
     * @pre !empty()
     * @throws std::underflow_error if the heap is empty.
     */
    [[nodiscard]] const T& max() const;

    /** @brief True if the heap contains no elements. */
    [[nodiscard]] bool empty() const;

    /** @brief Number of elements in the heap. */
    [[nodiscard]] int size() const;
};
```

Key Doxygen tags:

| Tag | Purpose |
|-----|---------|
| `@brief` | One-line summary |
| `@param` | Document a parameter |
| `@return` | Document the return value |
| `@tparam` | Document a template parameter |
| `@pre` | Precondition |
| `@post` | Postcondition |
| `@throws` | What exception is thrown and when |
| `@invariant` | Class invariant |
| `@note` | Additional note |
| `@warning` | Important warning |
| `@thread_safety` | Thread-safety guarantees (not standard but widely used) |

## Thread Safety Documentation

\index{thread safety!documentation}

Thread safety is one of the most important and hardest-to-test interface properties.
Be explicit about it:

```cpp
/**
 * @brief A thread-safe bounded queue.
 *
 * @thread_safety Thread-safe: enqueue and dequeue may be called
 *   concurrently from multiple threads.
 *
 * @thread_safety All operations use an internal mutex and are safe
 *   to call from any thread.
 */
class BoundedQueue { /* ... */ };

/**
 * @brief A non-thread-safe linked list.
 *
 * @thread_safety Not thread-safe. External synchronization is required
 *   if this list is accessed from multiple threads.
 *
 * @thread_safety Exception: `size()` and `empty()` are safe to call
 *   concurrently with each other (but not with mutating operations).
 */
template<typename T>
class LinkedList { /* ... */ };
```

If your data structure is not thread-safe, say so clearly.
Callers who need thread safety will wrap it in a mutex; callers who do not will avoid unnecessary overhead.

## Versioning and Backward Compatibility

\index{API!versioning}
\index{backward compatibility}

Public APIs change over time.
When users depend on your data structure, breaking changes cause compilation errors in their code --- even if your change was for the better.

### Additive Changes Are Safe

Adding new methods does not break existing callers:

```cpp
// Version 1.0
class Stack {
public:
    void push(const T& v);
    T pop();
    const T& top() const;
    bool empty() const;
    int size() const;
};

// Version 1.1: added clear() --- backward compatible
class Stack {
public:
    void push(const T& v);
    T pop();
    const T& top() const;
    bool empty() const;
    int size() const;
    void clear();  // new --- existing callers unaffected
};
```

### Breaking Changes to Avoid

- Removing a method.
- Changing a method's signature (parameter types, return type, `const`-ness).
- Adding a pure virtual method to a base class (breaks existing derived classes).
- Changing the semantics of an existing method (even without changing its signature).

### Deprecation

Use `[[deprecated]]` to warn callers of methods that will be removed:

```cpp
[[deprecated("use remove_max() instead")]]
T extract_max() { return remove_max(); }
```

Callers of `extract_max()` get a compiler warning with the migration message.
They can continue using it while they migrate, and you can remove it in a later version.

## Header-Only vs. Compiled Libraries

\index{header-only library}

Template data structures must be header-only: the full definition must be visible when the compiler instantiates the template.
Non-template data structures can be split into header (`.h`) and implementation (`.cpp`) files.

| Type | Header contains | `.cpp` contains |
|------|----------------|-----------------|
| Template class | Full definition | Nothing (or explicit instantiations) |
| Non-template class | Declaration | Definition |

For a library you want others to use:

```
// my_stack.h
#pragma once

template<typename T>
class Stack {
public:
    // Full implementation here
    void push(const T& v) { /* ... */ }
    // ...
};
```

::: {.tip}
**Tip:** Use `#pragma once` or include guards in every header file.
Multiple inclusions of the same header (via transitive includes) must not cause errors.
:::

3. `explicit` on the constructor.
4. A thread-safety comment at the class level.
5. A `[[deprecated]]` alias for a method you want to rename.

```bash

```

## Key Points

- **Type system encoding**: express constraints in types (use `explicit`, `[[nodiscard]]`, `const`) rather than documentation alone.
- Single-argument constructors should be `explicit` to prevent silent implicit conversions.
- `[[nodiscard]]` turns "silently ignoring the return value" into a compiler warning.
- `const` member functions document and enforce that a function does not modify the object.
- **Doxygen** generates API reference documentation from structured comments.
  Document `@pre`, `@post`, `@throws`, `@invariant`, and `@thread_safety` for every public method.
- **Thread safety** must be explicitly documented; callers cannot infer it from the code.
- **Additive changes** (new methods) are backward compatible; removing or renaming methods is a breaking change.
- Use `[[deprecated]]` to give callers a migration path before removing a method.
- Template classes must be header-only; non-template classes can be split into `.h` and `.cpp`.
