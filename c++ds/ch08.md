# 8. Hash Tables

\index{hash table}

Arrays give you O(1) access *if you know the index*.
Hash tables give you O(1) access *by key* --- whether the key is a string, an integer, or anything else you can hash.
They are the data structure behind `std::unordered_map`, `std::unordered_set`, Python dicts, and countless databases.
In this chapter you will learn how hash functions work, how to handle collisions, how load factor affects performance, and how to implement a hash table in C++.

## The Hash Table ADT

\index{hash table!ADT}

A hash table maps **keys** to **values**.
It supports:

| Operation | Description | Average complexity |
|-----------|-------------|-------------------|
| `insert(key, value)` | Associate value with key | O(1) |
| `lookup(key)` | Return value for key | O(1) |
| `remove(key)` | Remove the key-value pair | O(1) |
| `contains(key)` | True if key is present | O(1) |

All average-case O(1) --- with occasional O(n) worst case due to collisions.

## Hash Functions

\index{hash function}

A **hash function** maps a key to an integer index in the table.
The ideal hash function:

- Is deterministic: the same key always produces the same index.
- Is fast to compute.
- Distributes keys uniformly across the table to minimize collisions.

```cpp
#include <iostream>
#include <string>

// A simple hash for strings
std::size_t simple_hash(const std::string& key, std::size_t table_size)
{
    std::size_t hash = 0;
    for (char c : key) {
        hash = hash * 31 + static_cast<unsigned char>(c);
    }
    return hash % table_size;
}

int main()
{
    std::size_t size = 11;
    std::cout << simple_hash("Moby Dick", size) << "\n";
    std::cout << simple_hash("Billy Budd", size) << "\n";
    std::cout << simple_hash("Typee", size) << "\n";

    return 0;
}
```

```
3
7
2
```

The multiplier 31 is a common choice because it spreads bits well and multiplication by a small prime tends to produce good distributions.

::: {.tip}
**Tip:** Never write your own hash function for production code.
Use `std::hash<T>` (the standard library's hash) or a well-studied algorithm like FNV-1a or MurmurHash3.
Roll-your-own hash functions have subtle collision patterns that degrade performance in practice.
:::

## Collision Resolution Strategies

\index{collision}
\index{collision resolution}

A **collision** occurs when two different keys hash to the same index.
No matter how good your hash function is, collisions are inevitable (by the pigeonhole principle) once you store more than `table_size` items.
The two main resolution strategies are **separate chaining** and **open addressing**.

## Separate Chaining

\index{separate chaining}

In **separate chaining**, each table slot holds a list (chain) of key-value pairs.
Collisions just add to the list at that slot.

```
index 0: -> [Moby Dick, "Ishmael"] -> nullptr
index 1: -> nullptr
index 2: -> [Typee, "Tommo"] -> nullptr
index 3: -> [Billy Budd, "Sailor"] -> nullptr
...
```

```cpp
#include <functional>
#include <iostream>
#include <list>
#include <stdexcept>
#include <string>
#include <utility>
#include <vector>

template<typename K, typename V>
class ChainedHashTable {
public:
    explicit ChainedHashTable(int capacity = 16)
        : buckets_(capacity), size_{0}
    {}

    void insert(const K& key, const V& value)
    {
        auto& bucket = get_bucket(key);
        for (auto& [k, v] : bucket) {
            if (k == key) {
                v = value;  // update existing
                return;
            }
        }
        bucket.emplace_back(key, value);
        ++size_;
    }

    V& lookup(const K& key)
    {
        auto& bucket = get_bucket(key);
        for (auto& [k, v] : bucket) {
            if (k == key) return v;
        }
        throw std::out_of_range("key not found");
    }

    const V& lookup(const K& key) const
    {
        const auto& bucket = get_bucket(key);
        for (const auto& [k, v] : bucket) {
            if (k == key) return v;
        }
        throw std::out_of_range("key not found");
    }

    bool contains(const K& key) const
    {
        const auto& bucket = get_bucket(key);
        for (const auto& [k, v] : bucket) {
            if (k == key) return true;
        }
        return false;
    }

    void remove(const K& key)
    {
        auto& bucket = get_bucket(key);
        for (auto it = bucket.begin(); it != bucket.end(); ++it) {
            if (it->first == key) {
                bucket.erase(it);
                --size_;
                return;
            }
        }
        throw std::out_of_range("key not found");
    }

    int size() const { return size_; }
    bool empty() const { return size_ == 0; }

private:
    std::list<std::pair<K, V>>& get_bucket(const K& key)
    {
        std::size_t idx = std::hash<K>{}(key) % buckets_.size();
        return buckets_[idx];
    }

    const std::list<std::pair<K, V>>& get_bucket(const K& key) const
    {
        std::size_t idx = std::hash<K>{}(key) % buckets_.size();
        return buckets_[idx];
    }

    std::vector<std::list<std::pair<K, V>>> buckets_;
    int size_;
};

int main()
{
    ChainedHashTable<std::string, std::string> table;
    table.insert("Moby Dick", "Ishmael");
    table.insert("Billy Budd", "Sailor");
    table.insert("Typee", "Tommo");

    std::cout << table.lookup("Moby Dick") << "\n";
    std::cout << table.lookup("Typee") << "\n";
    std::cout << "Contains Billy Budd: " << table.contains("Billy Budd") << "\n";

    table.remove("Billy Budd");
    std::cout << "After remove, contains Billy Budd: "
              << table.contains("Billy Budd") << "\n";

    return 0;
}
```

```
Ishmael
Tommo
Contains Billy Budd: 1
After remove, contains Billy Budd: 0
```

## Open Addressing

\index{open addressing}
\index{linear probing}

In **open addressing**, all entries are stored in the table itself --- no separate lists.
When a collision occurs, you probe for the next available slot.

**Linear probing** tries slots `h`, `h+1`, `h+2`, ... until it finds an empty one:

```cpp
#include <functional>
#include <iostream>
#include <optional>
#include <stdexcept>
#include <string>
#include <vector>

template<typename K, typename V>
class OpenHashTable {
public:
    explicit OpenHashTable(int capacity = 16)
        : slots_(capacity), size_{0}, capacity_{capacity}
    {}

    void insert(const K& key, const V& value)
    {
        if (static_cast<double>(size_ + 1) / capacity_ > 0.7) {
            rehash();
        }
        int idx = probe(key);
        if (!slots_[idx]) {
            ++size_;
        }
        slots_[idx] = {key, value};
    }

    V& lookup(const K& key)
    {
        int start = static_cast<int>(std::hash<K>{}(key) % capacity_);
        for (int i = 0; i < capacity_; ++i) {
            int idx = (start + i) % capacity_;
            if (!slots_[idx]) break;
            if (slots_[idx]->first == key) return slots_[idx]->second;
        }
        throw std::out_of_range("key not found");
    }

    bool contains(const K& key) const
    {
        int start = static_cast<int>(std::hash<K>{}(key) % capacity_);
        for (int i = 0; i < capacity_; ++i) {
            int idx = (start + i) % capacity_;
            if (!slots_[idx]) return false;
            if (slots_[idx]->first == key) return true;
        }
        return false;
    }

    int size() const { return size_; }

private:
    int probe(const K& key)
    {
        int start = static_cast<int>(std::hash<K>{}(key) % capacity_);
        for (int i = 0; i < capacity_; ++i) {
            int idx = (start + i) % capacity_;
            if (!slots_[idx] || slots_[idx]->first == key) return idx;
        }
        throw std::overflow_error("table is full");
    }

    void rehash()
    {
        int new_cap = capacity_ * 2;
        std::vector<std::optional<std::pair<K, V>>> new_slots(new_cap);
        for (auto& slot : slots_) {
            if (slot) {
                int start = static_cast<int>(std::hash<K>{}(slot->first) % new_cap);
                for (int i = 0; i < new_cap; ++i) {
                    int idx = (start + i) % new_cap;
                    if (!new_slots[idx]) {
                        new_slots[idx] = std::move(slot);
                        break;
                    }
                }
            }
        }
        slots_ = std::move(new_slots);
        capacity_ = new_cap;
    }

    std::vector<std::optional<std::pair<K, V>>> slots_;
    int size_;
    int capacity_;
};

int main()
{
    OpenHashTable<std::string, int> table;
    table.insert("Moby Dick", 1851);
    table.insert("Billy Budd", 1924);
    table.insert("Typee", 1846);

    std::cout << "Moby Dick: " << table.lookup("Moby Dick") << "\n";
    std::cout << "Typee: " << table.lookup("Typee") << "\n";
    std::cout << "Contains Billy Budd: " << table.contains("Billy Budd") << "\n";

    return 0;
}
```

```
Moby Dick: 1851
Typee: 1846
Contains Billy Budd: 1
```

::: {.tip}
**Trap:** Deleting from an open-addressing hash table is tricky.
If you simply clear a slot, you break the probe chains: a later lookup may stop at the cleared slot before finding the key it was looking for.
The standard fix is to mark deleted slots with a "tombstone" sentinel rather than clearing them.
:::

## Load Factor and Rehashing

\index{load factor}
\index{rehashing}

The **load factor** α = n / capacity, where n is the number of stored items.

- When α is low (< 0.5), few collisions occur and operations are fast.
- When α approaches 1 (for open addressing) or exceeds 1 (for separate chaining), performance degrades.

**Rehashing** allocates a larger table (typically 2× the capacity) and reinserts all existing entries.
This is O(n) but happens infrequently enough that the amortized cost per insertion is O(1).

| Strategy | Typical rehash threshold | Behavior at high load |
|----------|--------------------------|-----------------------|
| Separate chaining | α > 1 | Chains grow; O(n) worst case per op |
| Open addressing | α > 0.5-0.7 | Clustering; probe sequences grow long |

::: {.tip}
**Tip:** `std::unordered_map` rehashes when the load factor exceeds `max_load_factor()` (default 1.0).
You can call `reserve(n)` to pre-allocate enough buckets for n elements and avoid rehashing.
:::

## Performance Analysis

\index{hash table!performance}

| Operation | Average | Worst case |
|-----------|---------|-----------|
| `insert` | O(1) | O(n) |
| `lookup` | O(1) | O(n) |
| `remove` | O(1) | O(n) |
| `contains` | O(1) | O(n) |
| **Space** | O(n + capacity) | O(n + capacity) |

Worst case occurs when all keys hash to the same bucket (degenerate hash function or adversarial input).
With a good hash function and reasonable load factor, average-case O(1) is reliable in practice.

### Hash Tables vs. Binary Search Trees

| | Hash table | BST (`std::map`) |
|--|-----------|-----------------|
| Lookup | O(1) average | O(log n) |
| Ordered iteration | No | Yes |
| Worst case | O(n) | O(log n) (balanced) |
| Key requirements | Hashable | Comparable |

Use a hash table (`std::unordered_map`) when you need fast lookup and do not need keys in sorted order.
Use a BST (`std::map`) when you need sorted iteration or range queries.

## Try It: Word Frequency Counter

```cpp
#include <algorithm>
#include <iostream>
#include <sstream>
#include <string>
#include <unordered_map>
#include <vector>

int main()
{
    std::string text = "to be or not to be that is the question to be";
    std::istringstream iss{text};
    std::unordered_map<std::string, int> freq;

    std::string word;
    while (iss >> word) {
        ++freq[word];
    }

    // Sort by frequency descending
    std::vector<std::pair<std::string, int>> pairs(freq.begin(), freq.end());
    std::sort(pairs.begin(), pairs.end(),
              [](const auto& a, const auto& b) { return a.second > b.second; });

    for (const auto& [w, count] : pairs) {
        std::cout << w << ": " << count << "\n";
    }

    return 0;
}
```

```
to: 3
be: 3
or: 1
not: 1
that: 1
is: 1
the: 1
question: 1
```

## Key Points

- A **hash table** maps keys to values with average O(1) insert, lookup, and remove.
- A **hash function** maps a key to a table index; good functions distribute keys uniformly.
- **Collisions** are inevitable; they are handled by separate chaining or open addressing.
- **Separate chaining:** each bucket holds a list of entries.
  Simple but uses extra memory per node.
- **Open addressing (linear probing):** collisions probe to the next available slot.
  Better cache locality but sensitive to load factor.
- **Load factor** α = n / capacity; rehash (usually 2× capacity) when load is too high.
- Deletion in open addressing requires tombstone markers to preserve probe chains.
- Use `std::unordered_map`/`std::unordered_set` in production; they are heavily optimized.
- Use `std::map` when you need sorted order; hash tables do not maintain it.
